thcode uses a bring-your-own-key (BYOK) model: you supply your own API keys for each provider, and those keys stay entirely on your machine. The credential store is the single gateway through which secrets are written and retrieved. No key value ever appears in a config file, `.env`, log, telemetry payload, prompt, or session transcript — the only place a key lives is the OS-managed credential facility.

## The `CredentialStore` Interface

Every platform adapter implements the same interface so the rest of thcode never needs to know how the OS stores secrets:

```typescript
interface CredentialStore {
  get(id: CredentialId): Promise<string | null>;
  set(id: CredentialId, secret: string): Promise<void>;
  delete(id: CredentialId): Promise<void>;
  has(id: CredentialId): Promise<boolean>;
  availability?(): CredentialStoreAvailability;
}
```

`CredentialId` is a plain string — typically the provider name (`"typhoon"`, `"aiforthai"`) or the reserved per-install session data key id (`"__thcode_session_dek__"`). The `availability()` probe is non-mutating: it returns the backend kind without reading or writing any secret.

```typescript
type CredentialStoreAvailability =
  | { kind: 'ok'; backend: 'windows-dpapi' | 'macos-keychain' | 'linux-secret-service' | 'in-memory' }
  | { kind: 'unavailable'; reason: string }
  | { kind: 'unknown'; reason: string };
```

## Windows DPAPI Backend

The Windows-first prototype stores each secret as a **DPAPI-encrypted file** under `%LOCALAPPDATA%\thcode\credentials\<id>.cred`. Encryption and decryption run through PowerShell 7 (`pwsh.exe`) using `ConvertFrom-SecureString` / `ConvertTo-SecureString` in the `CurrentUser` DPAPI scope. This approach:

- requires no native npm dependency;
- ties the ciphertext to the OS user account (another user on the same machine cannot decrypt it);
- keeps all storage logic behind the `CredentialStore` interface so platform adapters can be swapped.

<Note>
Secrets are piped to PowerShell over **stdin**, never over command-line arguments. This prevents the secret from appearing in process listings or shell history.
</Note>

### File locations

Platform paths follow OS conventions so session data, credentials, and checkpoints are stored in the correct user-state directory:

<Tabs>
  <Tab title="Windows">
    ```
    %LOCALAPPDATA%\thcode\
      credentials\          ← DPAPI-encrypted .cred files
      sessions.db           ← SQLite session database
    ```
    `%LOCALAPPDATA%` defaults to `C:\Users\<you>\AppData\Local` if the environment variable is unset.
  </Tab>
  <Tab title="macOS (deferred)">
    ```
    ~/Library/Application Support/thcode/
      credentials/          ← macOS Keychain adapter (not yet implemented)
      sessions.db
    ```
    macOS Keychain support is a planned extension point.
  </Tab>
  <Tab title="Linux (deferred)">
    ```
    $XDG_STATE_HOME/thcode/           ← or ~/.local/state/thcode/
      credentials/          ← Linux Secret Service adapter (not yet implemented)
      sessions.db
    ```
    The XDG state directory is used when `$XDG_STATE_HOME` is set; otherwise `~/.local/state`.
  </Tab>
</Tabs>

<Warning>
macOS Keychain and Linux Secret Service adapters are deferred — the current release ships a Windows-first prototype only. On macOS and Linux the credential store reports `kind: 'unavailable'` at the preflight probe.
</Warning>

## Onboarding Flow

The onboarding flow for a provider key follows a fixed sequence that ensures disclosure happens before the user ever types a secret.

<Steps>
  <Step title="Select a provider">
    The CLI presents the available providers. Typhoon is the only Release-1 reasoning provider; AI for Thai is an optional separate credential for specialist tools.
  </Step>
  <Step title="Disclosure">
    Before accepting any input, the CLI prints exactly where the credential will be stored and which host it will contact. For Typhoon:

    ```
    Typhoon is the only Release-1 reasoning provider.
    Your API key is stored in the OS credential facility (Windows Credential Manager / macOS Keychain).
    The key is never written to .env, project files, SQLite, journal, transcript, or output.
    AI-for-Thai credentials are separate and onboarded just-in-time.
    You can remove or rotate the key at any time.
    ```

    Onboarding is blocked entirely in non-interactive (headless) environments; it fails closed with a `headless-blocked` result rather than reading from stdin.
  </Step>
  <Step title="Masked key input">
    The key is accepted through a masked, non-echoing input prompt. Empty keys are rejected before any write to the credential store.
  </Step>
  <Step title="Write to OS credential store">
    The trimmed key is written via `store.set(providerId, trimmedKey)`. If the OS credential facility is unavailable, onboarding returns a typed `failed` result — it never falls through to a plaintext write.
  </Step>
  <Step title="Provider health check">
    After the key is stored, the CLI runs a live health probe against the configured endpoint to confirm the credential is valid and the provider is reachable. See [Typhoon health checks](/providers/typhoon#provider-health-checks) for probe details.
  </Step>
  <Step title="Display connection status">
    The CLI shows the provider identity, endpoint host, and connection state. The key value is never surfaced — only a secret-free fingerprint or generation id is shown.
  </Step>
</Steps>

## Security Invariants

The following rules are enforced throughout the codebase and must be preserved in any extension or integration:

<Accordion title="Never persist in repo, config, logs, or telemetry">
  A provider key must never appear in:
  - any file committed to a repository (`.env`, config YAML, JSON, TOML, etc.)
  - thcode config files
  - log output, crash reports, or telemetry payloads
  - prompts, tool results, or session transcripts
  - SQLite session database entries or journal files
</Accordion>

<Accordion title="Never send to thcode server or cross-provider">
  Each key travels only to its configured official provider endpoint:
  - The Typhoon key goes only to `api.opentyphoon.ai`.
  - The AI for Thai key goes only to `api.aiforthai.in.th`.
  - Neither key ever reaches the hosted thcode service.
  - A key for one provider is never reused against a different provider hostname.
</Accordion>

<Accordion title="Redact from tool output">
  Recognized credential patterns are redacted from tool output before they can enter a reasoning turn and appear in context.
</Accordion>

<Accordion title="Support explicit removal and rotation">
  Users can run `/disconnect` at any time to remove a stored credential. Removing or rotating a key invalidates the current configuration generation and puts dependent providers back into `unconfigured` state — no cached key value remains in any product buffer or persistence layer.
</Accordion>

## Credential Removal and Rotation

```
/connect        Add or replace a locally stored provider credential
/disconnect     Remove a provider credential from the OS credential store
/models         Show models available through configured local providers
/status         Show provider hostname and connection status without secrets
```

Rotation follows the same onboarding flow as initial setup — the new key overwrites the stored entry via `store.set()`, a new configuration generation is registered, and a fresh health check runs against the updated credential.
